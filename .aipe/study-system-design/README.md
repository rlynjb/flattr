# Study — System Design (flattr)

A per-repo system-design guide for **flattr** — a grade-aware router for
self-powered travel, "optimized for flat, not fast." Generated by
`/aipe:study-system-design` in the two-pass audit shape: one lens audit plus
the architectural patterns this repo actually exercises.

The defining architecture: a **build-time / runtime split** with no backend and
no database. A Node pipeline turns OSM + elevation into a static `graph.json`;
an Expo / React Native app reads it and routes locally — and re-runs the same
pipeline *on the device* to cover areas past the prebuilt slice.

## Reading order

1. **`00-overview.md`** — the whole system in one diagram + a legend of every
   component. Read this first; if you read nothing else, read this.
2. **`audit.md`** — Pass 1. The 8 system-design lenses walked against flattr,
   with `not yet exercised` named honestly. The map of what's here.
3. The five pattern files (Pass 2), in order — each a deep dive into one
   load-bearing architectural decision:

   | File | Pattern | The capability it buys |
   |---|---|---|
   | `01-build-time-graph-artifact.md` | static artifact / SSG-for-a-graph | no backend, no DB |
   | `02-on-device-pipeline-rerun.md` | ISR-for-a-graph | coverage past the prebuilt slice |
   | `03-tile-merge-stitch.md` | graph composition + seam stitching | cross-tile routing |
   | `04-honest-fallback-routing.md` | soft constraints / fail-open + flag | "no flat route" ≠ "no route" |
   | `05-elevation-provider-fallback.md` | provider abstraction + persistent cache | routes survive a dead/throttled API |

## How the patterns relate

```
  the five patterns, as one story

  01 build-time artifact ──── the base case: compute once, ship JSON
        │
        │ same pipeline, run on the phone for new areas
        ▼
  02 on-device re-run ─────── extends coverage past the base
        │ produces separate graph tiles
        ▼
  03 tile merge & stitch ──── glues tiles into one routable graph
        │ which the router crosses
        ▼
  04 honest fallback routing ─ best flat route available, truthfully labeled
        ▲
        │ elevation (the only un-computable input) feeds grades
  05 elevation fallback + cache ─ keeps grades flowing when the API isn't
```

## Cross-links to neighboring guides

System-design owns *architectural boundaries and tradeoffs*. Mechanism-level
detail belongs to the foundation guides:

- **study-data-modeling** — the `Node` / `Edge` schema that `graph.json`
  serializes (the *shape* of the persisted data).
- **study-database-systems** — AsyncStorage key-value durability; the
  storage-engine internals beneath the elevation cache.
- **study-dsa-foundations** — A*, the admissible haversine heuristic, the
  hand-rolled binary heap (`pqueue.ts`), graph adjacency and traversal.
- **study-distributed-systems** — `not yet exercised` here (single user, single
  device); coordination-under-partial-failure lives there if flattr ever grows
  a sync/replication story.
- **study-runtime-systems** — the single-slot build serializer (`busyRef` +
  debounce + drain) as a concurrency-control mechanism on the device.
- **study-networking** — Overpass / Open-Meteo retry, backoff, and batching on
  the wire (`elevation.ts`, `overpass.ts`).

## Grounding

Every claim in these files is anchored to a real `file:line` in the flattr repo.
Inferred scale/production behavior is labeled as inference. Where a lens finds
nothing, the audit says `not yet exercised` rather than inventing architecture.
