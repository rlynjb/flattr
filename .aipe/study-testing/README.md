# Study — Testing & Correctness (flattr)

> How do you KNOW the router works — and will keep working after the next
> change? This folder answers that for the flattr codebase, grounded in the
> real `*.test.ts` files. Generated by `/aipe:study-testing`.

The through-line: a good suite tells you what a change broke before your
users do. flattr's suite is one of the strongest surfaces in the repo —
**130 tests, 22 files, 282ms, zero network, all green** (`npx vitest run`,
2026-06-19). The interesting part isn't the count; it's the *design*: a
shortest-path engine that has no business being trusted on vibes, pinned by
an algorithmic oracle and property tests.

---

## The seam that organizes this whole folder: determinism

```
  Where a finding belongs — the deterministic / eval split

  ┌─ study-testing (HERE) ───────────────────────────────────┐
  │  given known input, assert known output.                 │
  │  "astar cost EQUALS dijkstra cost" — exact.              │
  │  unit · integration · property · oracle.                 │
  └──────────────────────────────────────────────────────────┘
  ┌─ study-ai-engineering (NOT here) ────────────────────────┐
  │  evals — is a NON-deterministic output good enough /     │
  │  did it regress? LLM-as-judge, eval sets.                │
  └──────────────────────────────────────────────────────────┘
```

flattr sits entirely on the left. There is no LLM in the engine, the
pipeline, or the tests — the AI-eval seam is `not yet exercised`. Every
assertion in this repo is `expect(actual).toBe(expected)` or
`toBeCloseTo` — deterministic correctness, full stop. That's the right call
for a routing engine: the cost of a path is a number, not an opinion.

If flattr later grows an LLM feature (say, natural-language "find me a flat
loop near here"), the *prompt assembly and output parsing* would still be
deterministic and testable here; only the model output itself would hand off
to `study-ai-engineering`'s evals.

## The top findings (ranked)

1. **Best test:** the Dijkstra-vs-A* optimality oracle
   (`features/routing/astar.test.ts:37-53`). A* is only worth anything if it
   returns the *same optimal cost* as the uninformed baseline — this test is
   the gate that proves the heuristic stayed admissible.
   → `01-optimality-oracle.md`

2. **Biggest coverage gap:** the entire `mobile/src/` Expo app
   (`MapScreen.tsx`, `useTileGraph.ts`, `loadGraph.ts`, `AddressBar.tsx`,
   and three more) has **zero tests**. The engine it consumes is bulletproof;
   the UI that drives it is `not yet exercised`. → `audit.md` §1.

3. **Flakiness risk:** effectively none. No test touches the clock, the
   network, the filesystem, or shared mutable state. The one place real time
   *could* leak in — Open-Meteo's retry backoff — is defused by injecting
   `delayMs: 0` (`pipeline/elevation.test.ts:58,68`). → `audit.md` §4.

4. **No CI, no coverage tooling.** `npm test` is green locally but nothing
   runs it on push. → `audit.md` §7.
