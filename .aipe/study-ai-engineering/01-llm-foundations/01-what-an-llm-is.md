# What an LLM is

**Industry name(s):** large language model / autoregressive next-token
predictor / foundation model. **Type:** Industry standard primitive.

## Zoom out — where this would sit in flattr

An LLM is a pure function `tokens → tokens`: given a prefix of text, it
emits a probability distribution over the next token, samples one, appends
it, repeats. flattr's router is the *opposite kind of function* —
`directedAstar(graph, start, end, userMax) → path` is a deterministic
graph search with a provable answer. Same word, "function"; one is a
learned approximation, the other is exact code. The only place an LLM
would ever attach is the *prose* boundary, where numbers become a
sentence.

```
  Zoom out — deterministic core vs. the one place a learned function fits

  ┌─ Core engine (features/routing/) ───────────────────────┐
  │  directedAstar() ──► Path   (EXACT, provable, no model)  │
  │  routeSummary()  ──► RouteSummary {distanceM,climbM,     │
  │                                    steepCount}           │ ← deterministic
  └───────────────────────────┬─────────────────────────────┘
                              │ summary.ts:5  (OUTPUT→PROMPT seam)
  ┌─ UI (mobile/) ────────────▼─────────────────────────────┐
  │  RouteSummaryCard renders numbers as text (templated)    │
  │                                                          │
  │  ★ IF prose were generated by a model, THIS is the only  │
  │    spot an LLM (tokens→tokens) would live in flattr      │
  └──────────────────────────────────────────────────────────┘
```

flattr has **no LLM** anywhere — no weights, no inference call, no API
key. This file teaches what an LLM *is* so you can see precisely why the
router isn't one, and where the one would-be model would clip on.

## Structure pass

- **Layers:** engine (exact search) → summary (typed totals) → UI
  (renders text).
- **Axis — determinism:** the engine is fully deterministic — same
  inputs, same path, byte-for-byte. An LLM is *probabilistic* — same
  prompt can give different tokens unless you pin sampling. The axis
  flips from "exact" to "sampled."
- **Seam:** the flip would happen at `summary.ts:5` → `MapScreen.tsx:368`.
  Everything above that seam is provable; a route-describe LLM placed
  *below* it would be the first non-deterministic component in the app.

## How it works

### Move 1 — the mental model

You already know a hash map: key in, value out, deterministic. An LLM is
a *fuzzy, enormous, learned* lookup — but instead of one value, it
returns a distribution over the next token, and you sample from it. It
has no router, no graph, no ground truth; it predicts plausible text.

```
  Pattern — an LLM is one function applied in a loop

  prompt tokens  ──►┌────────────────────────┐
  [t0 t1 t2]        │ model: tokens → logits │
                    │ (billions of weights)  │
                    └───────────┬────────────┘
                                ▼
                       P(next token) distribution
                                │  sample one (t3)
                                ▼
  [t0 t1 t2 t3] ──► feed back in ──► repeat until <stop>
```

### Move 2 — the walkthrough

**flattr's function is the wrong shape to be an LLM.** The router:

```ts
// directedAstar(graph, startId, endId, userMax) → { path }
const r = directedAstar(graph, startId, endId, userMax);   // MapScreen.tsx:155
```

Inputs are a graph and two node ids; output is a *provably optimal* path
under the cost function. There is no distribution, no sampling, no
training. Calling this "AI" would be wrong — it's Dijkstra's
great-grandchild.

```
  Layers-and-hops — exact search, then the only learnable hop

  ┌─ engine ──┐ hop1: Path (exact)   ┌─ summary ──┐
  │directedAstar│ ──────────────────► │routeSummary│
  └───────────┘                      └─────┬──────┘
                                  RouteSummary {3 numbers}
  ┌─ UI ──────┐ hop2: text          ┌─ ★ would-be LLM ─┐
  │SummaryCard│ ◄────────────────── │ describeRoute()  │ ← tokens→tokens
  └───────────┘                     └──────────────────┘
```

**Where the model would attach.** At `summary.ts:11`, `routeSummary()`
returns `{ distanceM, climbM, steepCount }`. A describe-route LLM would
take those three numbers as a prompt and emit a sentence —
`MapScreen.tsx:159` produces the summary, `:368` consumes it. The model
slots between. That is the *entire* surface where "tokens→tokens" has a
home in this codebase.

**Why not classify.ts?** `classify.ts` looks model-ish — it maps a grade
to a color band — but it is an `if/else` over `DEFAULT_BANDS`
(`classify.ts:8`). It is a threshold table, not a learned function. Never
call it ML. (See [07-heuristic-before-llm.md](07-heuristic-before-llm.md).)

### Move 3 — the principle

An LLM is a sampled approximation of "what text comes next"; flattr's
router is an exact solver. Knowing the difference tells you where a model
*can't* go (the routing math — you need admissibility and proofs there)
and where it *can* (turning a correct answer into readable prose).

## Elaborate

The useful frame for a frontend/DSA engineer: an LLM is a function with
*no spec* — you can't write a unit test that asserts the exact output,
only that it satisfies properties (valid JSON, mentions the climb, etc).
That is why every other file in this section exists: tokenization,
sampling, streaming, economics, provider abstraction all manage the fact
that this one function is learned, not specified. In Rein's portfolio,
AdvntrCue (GPT-4) and dryrun (on-device Gemini Nano) are real instances
of this function; flattr is the deterministic foil that shows where it
would and wouldn't belong.

## Project exercises

### B-LLM.1 — name the boundary in code

- **Exercise ID:** B-LLM.1
- **What to build:** a stub `describeRoute(summary: RouteSummary): string`
  in a new file that returns a templated sentence (no model), so the
  tokens→tokens boundary exists as a named function before any inference.
- **Why it earns its place:** it makes "where an LLM would go" a real
  call site instead of a hand-wave, and keeps the engine deterministic.
- **Files to touch:** new `features/routing/describe.ts`;
  `mobile/src/MapScreen.tsx:159` (call it next to `routeSummary`).
- **Done when:** the stub compiles, has a unit test, and the engine
  output is unchanged.
- **Estimated effort:** 1 hr.

### B-LLM.2 — determinism contrast test

- **Exercise ID:** B-LLM.2
- **What to build:** a test that calls `directedAstar` twice with the
  same inputs and asserts byte-identical paths, documenting that the core
  is deterministic (the property an LLM lacks).
- **Why it earns its place:** it pins the exact property that disqualifies
  the router from being a model and that a future LLM hop must respect.
- **Files to touch:** `features/routing/` test file; `cost.ts` (inputs).
- **Done when:** the test passes and references the seam in a comment.
- **Estimated effort:** 30 min.

## Interview defense

**Q: Is flattr's router AI?** Answer: No. It's exact graph search —
`directedAstar` returns a provably optimal path under a cost function;
same inputs always give the same path. An LLM is a sampled tokens→tokens
function with no spec. The only place a model fits in flattr is turning
`RouteSummary`'s three numbers into prose, at the `summary.ts:5` seam —
and that would be the first non-deterministic component in the app.

```
  exact: graph → directedAstar → Path        (today)
  sampled: {3 numbers} → [LLM] → sentence     (NOT built; summary.ts:5)
```

Anchor: *"flattr's router is an exact solver; an LLM is a sampled
approximation — they meet only at the prose seam in summary.ts."*
